<h2 id="intro">Qué es y para qué sirve</h2>

`react-rescuer` es una librería para manejar errores en React usando un `ErrorBoundary` (class-based) con:

- Recuperación: UI de fallback con `resetError()` y (opcionalmente) reintentos controlados.
- Contexto estructurado: cuando ocurre un error, se construye un `ErrorContext` con fingerprint, breadcrumbs, component stack, session id, etc.
- Observabilidad opcional: breadcrumbs y fingerprinting desacoplados del core (se integran por inyección de funciones).
- Reporting: un hook `onError(error, errorInfo, errorContext)` para enviar a cualquier herramienta.

<h2 id="key-concepts">Conceptos clave</h2>

### 1) Error boundaries solo atrapan errores de render

- Atrapan errores que ocurren durante render, lifecycle y constructors en el árbol hijo.
- No atrapan errores de handlers/eventos o async por defecto.

### 2) Para errores async/eventos, usa `useErrorBoundary()`

- Este hook permite "elevar" un error al boundary más cercano llamando `showBoundary(error)`.

### 3) Cuando un boundary atrapa un error, crea un `ErrorContext`

`ErrorContext` es el objeto que se pasa a fallbacks y al callback `onError`.

### 4) Reset y recovery

- `resetError()` resetea el boundary.
- `resetKeys` resetea automáticamente cuando cambia el array.
- `recovery` habilita reintentos con límite y delay/backoff.

<h2 id="quick-start">Guía rápida (30-60 segundos)</h2>

### 1) Envuelve una ruta/sección

```tsx
import { ErrorBoundary } from "react-rescuer";

export const App = () => {
  return (
    <ErrorBoundary fallback={<div>Algo salió mal.</div>}>
      <Page />
    </ErrorBoundary>
  );
};
```

### 2) Usa un fallback real con botón de reset

```tsx
import { ErrorBoundary } from "react-rescuer";
import type { FallbackProps } from "react-rescuer";

const Fallback = ({ error, resetError }: FallbackProps) => {
  return (
    <div role="alert">
      <div>Oops:</div>
      <pre>{error.message}</pre>
      <button type="button" onClick={resetError}>
        Reintentar
      </button>
    </div>
  );
};

export const App = () => {
  return (
    <ErrorBoundary FallbackComponent={Fallback}>
      <Page />
    </ErrorBoundary>
  );
};
```

### 3) (Opcional) Reporta el error con contexto

```tsx
import { ErrorBoundary } from "react-rescuer";

export const App = () => {
  return (
    <ErrorBoundary
      onError={(error, _info, ctx) => {
        // Envía a Sentry/Datadog/tu API
        console.log("caught", { error, ctx });
      }}
      fallback={<div>Algo salió mal.</div>}
    >
      <Page />
    </ErrorBoundary>
  );
};
```

<h2 id="error-boundary">ErrorBoundary</h2>

`ErrorBoundary` es el centro de la librería.

### Modos de fallback (elige uno)

#### 1) `fallback`: un ReactNode estático

```tsx
<ErrorBoundary fallback={<div>Fallback simple</div>}>
  <Page />
</ErrorBoundary>
```

#### 2) `fallbackRender`: render prop (recibe props)

```tsx
import { ErrorBoundary } from "react-rescuer";

<ErrorBoundary
  fallbackRender={({ error, errorContext, resetError, retryCount }) => (
    <div>
      <div>{error.message}</div>
      <div>fingerprint: {errorContext.fingerprint}</div>
      <div>retryCount: {retryCount}</div>
      <button type="button" onClick={resetError}>
        Retry
      </button>
    </div>
  )}
>
  <Page />
</ErrorBoundary>;
```

#### 3) `FallbackComponent`: componente de fallback

```tsx
import { ErrorBoundary } from "react-rescuer";
import type { FallbackProps } from "react-rescuer";

const FallbackUI = ({ error, resetError }: FallbackProps) => {
  return (
    <div>
      <div>Error: {error.message}</div>
      <button onClick={resetError}>Reset</button>
    </div>
  );
};

<ErrorBoundary FallbackComponent={FallbackUI}>
  <Page />
</ErrorBoundary>;
```

### Qué ocurre internamente cuando se atrapa un error

Cuando un hijo falla durante render:

1. React llama `componentDidCatch(error, errorInfo)`.
2. El boundary incrementa su contador interno `errorCount`.
3. Construye `boundaryProps` (snapshot de configuración) y genera un `ErrorContext`.
4. Llama `onError?.(error, errorInfo, errorContext)`.
5. Renderiza el fallback elegido (y provee `ErrorContext` vía React context para `useErrorContext`).

### API (tipos principales)

Estos son los tipos reales del paquete (resumen fiel a `packages/lib/src/core/types.ts`).

```ts
export type ErrorBoundaryLevel = "page" | "section" | "component";
export type BreadcrumbType = "click" | "navigation" | "custom";

export type Breadcrumb = {
  type: BreadcrumbType;
  timestamp: number;
  message?: string;
  data?: Record<string, unknown>;
};

export type ErrorContext = {
  error: Error;
  fingerprint: string;
  breadcrumbs: Breadcrumb[];
  componentStack: string;
  timestamp: number;
  sessionId: string;
  errorCount: number;
  boundaryProps?: unknown;
};

export type FallbackProps<E extends Error = Error> = {
  error: E;
  errorContext: ErrorContext;
  resetError: () => void;
  retryCount: number;
};

export type RecoveryStrategy<E extends Error = Error> = {
  maxRetries: number;
  retryDelay?: number | ((attempt: number, error: E) => number);
  isRecoverable?: (error: E) => boolean;
  onMaxRetriesReached?: (error: E, context: ErrorContext) => void;
};

export type ResetReason = "imperative" | "resetKeys" | "retry";

export type ErrorBoundaryProps<E extends Error = Error> = {
  children: React.ReactNode;

  level?: ErrorBoundaryLevel;
  isolate?: boolean;
  resetKeys?: unknown[];

  FallbackComponent?: React.ComponentType<FallbackProps<E>>;
  fallbackRender?: (props: FallbackProps<E>) => React.ReactNode;
  fallback?: React.ReactNode;

  onError?: (
    error: E,
    errorInfo: React.ErrorInfo,
    errorContext: ErrorContext,
  ) => void;
  onReset?: (details: { reason: ResetReason }) => void;

  recovery?: RecoveryStrategy<E>;

  getBreadcrumbs?: () => Breadcrumb[];
  fingerprint?: (error: E) => string;
  contextBuilder?: (
    error: E,
    errorInfo: React.ErrorInfo,
    options: {
      sessionId: string;
      errorCount: number;
      retryCount: number;
      boundaryProps?: unknown;
      getBreadcrumbs?: () => Breadcrumb[];
      fingerprint?: (error: E) => string;
    },
  ) => ErrorContext;
};
```

Notas sobre `level` / `isolate`

- Existen como props y se incluyen dentro del `boundaryProps` que termina dentro de `ErrorContext`.
- Hoy no cambian el comportamiento de captura/reset; puedes usarlos como metadata (por ejemplo para analítica o overlays).

<h2 id="resets">Resets</h2>

### Reset manual: `resetError()`

- `resetError()` viene en los props del fallback.
- Si no configuraste `recovery`, `resetError()` hace reset inmediato.
- Si configuraste `recovery`, `resetError()` puede convertirse en "intento de recovery".

### Reset automático: `resetKeys`

Cuando `resetKeys` cambia (comparación por elemento con `Object.is`), el boundary resetea con reason `"resetKeys"`.

```tsx
import { useState } from "react";
import { ErrorBoundary } from "react-rescuer";

const Bomb = ({ armed }: { armed: boolean }) => {
  if (armed) throw new Error("boom");
  return <div>OK</div>;
};

export const Demo = () => {
  const [armed, setArmed] = useState(false);

  return (
    <div>
      <button onClick={() => setArmed(true)}>Throw</button>
      <button onClick={() => setArmed(false)}>Reset</button>

      <ErrorBoundary resetKeys={[armed]} fallback={<div>Fallback</div>}>
        <Bomb armed={armed} />
      </ErrorBoundary>
    </div>
  );
};
```

### Evento de reset para integraciones: `react-rescuer:reset`

En cada reset (imperative, resetKeys, retry), el boundary intenta disparar un evento DOM:

- Event name: `react-rescuer:reset`
- `detail`: `{ sessionId }`

Esto se usa, por ejemplo, para limpiar breadcrumbs automáticamente.

<h2 id="recovery">Recovery / reintentos (`recovery`)</h2>

`recovery` es opcional. Si lo provees, `resetError()` aplica una política de reintentos:

- `maxRetries`: máximo número de intentos.
- `retryDelay`: delay antes de resetear. Puede ser:
  - `number`: delay fijo.
  - `function(attempt, error)`: delay dinámico.
- `isRecoverable`: si retorna `false`, `resetError()` hace reset inmediato (no cuenta como retry).
- `onMaxRetriesReached`: se llama cuando se supera el máximo; en ese caso NO se resetea (te quedas en fallback).

Ejemplo con backoff simple:

```tsx
import { ErrorBoundary } from "react-rescuer";

export const Demo = () => {
  return (
    <ErrorBoundary
      recovery={{
        maxRetries: 3,
        retryDelay: (attempt) => Math.min(1000, 200 * 2 ** (attempt - 1)),
      }}
      fallbackRender={({ error, resetError, retryCount }) => (
        <div>
          <div>{error.message}</div>
          <div>retryCount: {retryCount}</div>
          <button onClick={resetError}>Retry</button>
        </div>
      )}
    >
      <Page />
    </ErrorBoundary>
  );
};
```

Cómo se interpreta `retryCount`

- Empieza en `0`.
- Se incrementa cuando `resetError()` inicia un retry permitido.
- Se resetea a `0` cuando el reset es imperative o por `resetKeys`.

<h2 id="observability">Observabilidad (opcional)</h2>

La observabilidad está fuera del core; se integra con:

- `contextBuilder`
- o (`getBreadcrumbs` + `fingerprint`)

### 1) buildErrorContext (la opción más directa)

`buildErrorContext` arma un `ErrorContext` usando:

- breadcrumbs desde `getBreadcrumbTrail().get()` por defecto (o tu `getBreadcrumbs`)
- fingerprint desde `fingerprintError` por defecto (o tu `fingerprint`)
- component stack desde `errorInfo.componentStack`

```tsx
import { ErrorBoundary } from "react-rescuer";
import { addBreadcrumb, buildErrorContext } from "react-rescuer/observability";

export const Demo = () => {
  return (
    <ErrorBoundary
      contextBuilder={buildErrorContext}
      onError={(_err, _info, ctx) => {
        console.log("ErrorContext", ctx);
      }}
      fallback={<div>Mira la consola</div>}
    >
      <button
        type="button"
        onClick={() => addBreadcrumb({ type: "custom", message: "click" })}
      >
        Agregar breadcrumb
      </button>
    </ErrorBoundary>
  );
};
```

### 2) BreadcrumbTrail: breadcrumbs automáticos (click + navigation)

`BreadcrumbTrail` (singleton) captura:

- clicks (en capture) con un descriptor simple del elemento
- navegación (popstate y monkeypatch de `history.pushState`/`replaceState`)

También limpia la cola cuando ocurre un reset, escuchando `react-rescuer:reset`.

Uso:

```ts
import { getBreadcrumbTrail } from "react-rescuer/observability";

const trail = getBreadcrumbTrail();
trail.add({ type: "custom", message: "algo pasó" });

const breadcrumbs = trail.get();
```

`addBreadcrumb` es un helper:

```ts
import { addBreadcrumb } from "react-rescuer/observability";

addBreadcrumb({ type: "custom", message: "user clicked" });
```

### 3) Fingerprinting

`fingerprintError(error)` genera un hash determinístico (djb2) basado en:

- `error.name`
- las primeras 3 líneas del stack

Puedes proveer tu propio `fingerprint` si necesitas otra estrategia.

<h2 id="hooks">Hooks</h2>

### useErrorBoundary()

Para errores fuera de render (eventos, async, promesas):

```tsx
import { ErrorBoundary } from "react-rescuer";
import { useErrorBoundary } from "react-rescuer/hooks";

export const SaveButton = () => {
  const { showBoundary } = useErrorBoundary();

  return (
    <button
      type="button"
      onClick={async () => {
        try {
          await fetch("/api/save");
          throw new Error("error async");
        } catch (e) {
          showBoundary(e as Error);
        }
      }}
    >
      Save
    </button>
  );
};

export const Demo = () => {
  return (
    <ErrorBoundary fallback={<div>Fallback</div>}>
      <SaveButton />
    </ErrorBoundary>
  );
};
```

Cómo funciona

- `showBoundary(error)` guarda el error en state y en el siguiente render lo vuelve a lanzar (`throw`).
- Ese throw ocurre en render, así que el ErrorBoundary más cercano lo atrapa.

### useErrorContext()

Para leer el `ErrorContext` dentro de tu fallback.

```tsx
import { ErrorBoundary } from "react-rescuer";
import { useErrorContext } from "react-rescuer/hooks";

export const FallbackWithContext = () => {
  const ctx = useErrorContext();

  return (
    <div>
      <div>fingerprint: {ctx.fingerprint}</div>
      <div>sessionId: {ctx.sessionId}</div>
      <div>errorCount: {ctx.errorCount}</div>
      <pre>{ctx.componentStack}</pre>
    </div>
  );
};

export const Demo = () => {
  return (
    <ErrorBoundary FallbackComponent={FallbackWithContext}>
      <Page />
    </ErrorBoundary>
  );
};
```

Nota

- Si lo llamas fuera de un fallback de `ErrorBoundary`, lanza un error.
- Internamente usa un contexto global (vía `globalThis`) para evitar problemas si hay múltiples instancias del paquete.

<h2 id="hoc">HOC</h2>

`withErrorBoundary(Component, options)` envuelve un componente con un `ErrorBoundary`.

```tsx
import { withErrorBoundary } from "react-rescuer/hoc";

export const ProfilePage = () => {
  return <div>...</div>;
};

export const SafeProfilePage = withErrorBoundary(ProfilePage, {
  fallback: <div>Profile fallback</div>,
});
```

<h2 id="retry-manager">Recovery standalone (RetryManager)</h2>

Si quieres manejar reintentos fuera del boundary:

```ts
import { RetryManager, createExponentialBackoff } from "react-rescuer/recovery";

const mgr = new RetryManager(
  {
    maxRetries: 3,
    retryDelay: (attempt) => Math.min(1000, 200 * 2 ** (attempt - 1)),
  },
  createExponentialBackoff(250, 10_000),
);

const { ok, attempt, delayMs } = mgr.next(
  "boundary-id",
  new Error("boom"),
  context,
);
```

Comportamiento

- `next()` incrementa el attempt por `boundaryId`.
- Si supera `maxRetries`, llama `onMaxRetriesReached` y retorna `{ ok: false }`.
- `reset(boundaryId)` limpia el contador.

<h2 id="testing">Testing</h2>

### createTestBoundary()

`createTestBoundary()` retorna un wrapper `Boundary` para tests y funciones para inspeccionar lo capturado.

```tsx
import { createTestBoundary } from "react-rescuer/testing";

const tb = createTestBoundary();
const { Boundary, getLastError, getLastContext, reset } = tb;

// render(
//   <Boundary>
//     <Bomb />
//   </Boundary>
// )

// expect(getLastError()).toBeInstanceOf(Error);
// expect(getLastContext()?.fingerprint).toBeTruthy();

reset();
```

### Matchers

La librería puede registrar matchers personalizados en `expect`:

```ts
import { installMatchers } from "react-rescuer/testing";

installMatchers();
```

Matchers incluidos:

- `toHaveCaughtError()`
- `toHaveCaughtErrorMatching(pattern)`

Uso:

```ts
import { createTestBoundary, installMatchers } from "react-rescuer/testing";

installMatchers();

const tb = createTestBoundary();
// render con tb.Boundary...

// expect(tb).toHaveCaughtError();
// expect(tb).toHaveCaughtErrorMatching(/boom/);
```

<h2 id="dev-overlay">Dev overlay (desarrollo)</h2>

En `development` (según `process.env.NODE_ENV`), `ErrorBoundary` intenta renderizar un overlay integrado usando un `require()` condicional.

El overlay muestra:

- nombre y mensaje del error
- stack trace filtrado
- component stack
- breadcrumbs
- botón `retry`

En `production`, siempre se renderiza tu fallback configurado.

<h2 id="limitations">Notas y limitaciones</h2>

- Los boundaries no atrapan errores de eventos/async por defecto: usa `useErrorBoundary()`.
- Si configuras `recovery` y superas `maxRetries`, el boundary se queda en el fallback (y se dispara `onMaxRetriesReached`).
- BreadcrumbTrail modifica `history.pushState` y `history.replaceState` cuando se inicia; puedes `stop()` si no lo quieres.
